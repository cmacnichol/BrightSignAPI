# AI Agent Context: BrightSignAPI

This file (`agents.md`) serves as a permanent context primer for any future AI agents or developers interacting with the `BrightSignAPI` repository. Please read this document before making architectural changes or updating documentation.

## 1. Project Scope
`BrightSignAPI` is an object-oriented PowerShell module designed to interact with the BSN.cloud REST API. Its primary goal is to automate the deployment and provisioning of BrightSign digital signage players by wrapping OAuth2 client-credentials authentication and cloning/modifying "Golden Template" setup folders.

## 2. Key Architectural Decisions & Nuances

### `[System.Net.HttpWebRequest]` over `Invoke-RestMethod`
The BSN.cloud Provisioning API (specifically the `/provisioning/setups/tokens` endpoint) occasionally returns HTTP 302 redirects containing HTTPS-to-HTTP `Location` headers. Native PowerShell cmdlets (`Invoke-RestMethod` and `Invoke-WebRequest`) in both PS 5.1 and PS 7 contain a hardcoded security "bug" where they explicitly throw an `InvalidOperationException` upon encountering this downgrade, rendering them entirely incapable of interacting with BSN.cloud.

**CRITICAL:** Do *not* refactor the module to use `Invoke-RestMethod`. All API calls must flow through the private helper function `Invoke-BsnApiRequest`, which uses raw `.NET` `[System.Net.HttpWebRequest]` objects and safely handles the redirect streams without throwing terminating exceptions.

### Global Connection State
Authentication and network selection are globally persisted in `$script:BsnSession`. Cmdlets should rarely require credential parameters; instead, they should securely pull the Bearer token dynamically via `Get-BsnAuthHeader`.

### Template-Driven Provisioning
When provisioning new players (`New-BsnPlayer`), do not attempt to generate XML/JSON config payloads from scratch. We rely exclusively on "Golden Templates." The user exports a working presentation from BrightAuthor:connected to a folder. Our module simply copies that folder and surgically rewrites specific values (Name, Registration Token, MACs, Certificates) inside the `setup.json` meta-client object. 

### Pipeline Scoping
Cmdlets that accept arrays or pipeline input (e.g. `Import-Csv | New-BsnPlayer`) must tightly scope internal variables inside the `process {}` block. Native pipeline parameters (like `$RegistrationToken`) cannot be manually modified during execution, or else PowerShell will "cache" that modification across the entire pipeline run, resulting in every device receiving the exact same token.

## 3. Documentation Generation Process

The `docs/wiki` directory is a mirror of the GitHub Wiki (`.wiki.git` repository).

**DO NOT MANUALLY EDIT CMDLET MARKDOWN FILES.**
The documentation is automatically generated via `PlatyPS`. If a cmdlet's parameters, synopsis, or description need updating:
1. Open the cmdlet's `.ps1` source code file.
2. Edit the comment-based help block (e.g. `.DESCRIPTION`, `.PARAMETER`).
3. Safely regenerate the wiki files using the workflow below.

### WARNING: PlatyPS Custom File Collision
PlatyPS possesses a known bug where `Update-MarkdownHelp` will completely crash if it encounters non-cmdlet Markdown files (like `Getting-Started.md`). Furthermore, running `New-MarkdownHelp -Force` will falsely map and overwrite custom files with random cmdlet data.

To safely regenerate the wiki documentation:

```powershell
# 1. Temporarily move custom files OUT of the wiki folder
Move-Item C:\Git\BrightSignAPI\docs\wiki\Getting-Started.md C:\Git\BrightSignAPI\docs\ -Force
Move-Item C:\Git\BrightSignAPI\docs\wiki\Architecture-Overview.md C:\Git\BrightSignAPI\docs\ -Force
Move-Item C:\Git\BrightSignAPI\docs\wiki\Deployment-Walkthrough.md C:\Git\BrightSignAPI\docs\ -Force

# 2. Force regenerate the cmdlet markdown
Remove-Module BrightSignAPI -ErrorAction SilentlyContinue
Import-Module C:\Git\BrightSignAPI\src\BrightSignAPI\BrightSignAPI.psd1 -Force
Import-Module platyPS
New-MarkdownHelp -Module BrightSignAPI -OutputFolder C:\Git\BrightSignAPI\docs\wiki -Force

# 3. Move the custom files back IN
Move-Item C:\Git\BrightSignAPI\docs\Getting-Started.md C:\Git\BrightSignAPI\docs\wiki\ -Force
Move-Item C:\Git\BrightSignAPI\docs\Architecture-Overview.md C:\Git\BrightSignAPI\docs\wiki\ -Force
Move-Item C:\Git\BrightSignAPI\docs\Deployment-Walkthrough.md C:\Git\BrightSignAPI\docs\wiki\ -Force
```
